## 3. 検証方法（なぜ1キーずつ実施したか）

通常は「数字キーで発声 → 直後に Y(聞こえた)/N(無音) → R で JSON 保存」という流れを想定していたが、
後述のバグにより**1セッション内で複数キーを連続操作すると記録が破綻する**。そのため次の手順で実施した。

1. プログラムを起動する。
2. 数字キーを **1つだけ** 押し、発声の有無を耳で確認する。
3. プログラムを終了する。
4. 次の数字キーで 1〜3 を繰り返す（キーごとにプログラムを起動し直す）。

この「1キー1起動」方式により、キー混線の影響を受けずに各手段の可聴を個別に判定した。

## 4. 既知のバグ（記録が信頼できない理由）

### 症状
直前に押したキーが、次のキーを押したタイミングで読み上げられてしまう。
例: `1` を押す → `Y` を押す → `2` を押すと「ｙ（ワイ）」と読み上げる。

### 影響
- 1セッション内では「どのキーで何が鳴ったか」を切り分けられない。
- Y/N の可聴判定が直前の手段に正しく結び付かず、JSON の `audible` が当てにならない。

### 原因（解析）
- プログラム本体は **数字キーでしか発声しない**（`MainForm.OnKeyDown` → `SpeechMethods.Invoke`）。
  Y/N/S/R では発声処理を呼んでいない。したがって「ｙ」という発話はプログラムの出力ではなく、
  **PC-Talker 自身のキーエコー（押下キーの読み上げ）**である。
- 本命手段の `PCTKPReadW(text, priority=0, analyze=1)` は **priority=0＝非割り込み**で発声をキューに積む。
  そのため PC-Talker のキーエコー（直前の `Y` の「ｙ」など）が消化されないまま残り、
  次キー押下時のプログラム発話と前後して聞こえる。結果として「直前キーを読む」ように知覚される。

### 記録上の帰結
- JSON は **毎回上書き**で、`ProbeRecorder` は起動ごとに新規生成される。
  「1キー1起動」運用では、保存ファイルには**最後に実行した1キー分しか残らない**。
- 実際、現存する `results/pctk-speech-results.json` はキー1のみ `attempts=1`、
  キー2〜6 は `attempts=0`（＝そのセッションでは未押下）。
  これは 2・3・4 が発声しなかったことを意味するのではなく、
  **保存ファイルが最後のキー1実行回のものに過ぎない**ためである。

## 5. 未確定事項 / 今後の確認

- **キー1（SoundMessage）の可聴**: API は 1 を返すが、可聴は未確認（yEdit の既知所見では「無音」）。要再確認。
- **キー4（PCTKCGuide）の記録化**: 発声は確認できたが、署名は未確認のまま。
  正式に `available=true` をログへ残すには、安定動作する署名で再実行する必要がある。
- **キー5/6**: 本検証では未確認（5 は Tolk.dll 未配置のため発声し得ない）。
- **記録バグの修正**（任意）: 例えば
  - 発声手段を `priority=割り込み` 側に寄せる、または
  - Y/N/S/R を `e.SuppressKeyPress = true` 等でキーエコーを抑制する、
  - 1セッションで全キーを記録できるよう、保存時にタイムスタンプ別ファイルへ追記する、
  といった対策で「1キー1起動」を不要にできる可能性がある。
